# Архитектура приложения BBPlay (Flutter)

## Процесс создания и краткое резюме 

При создании приложения была использована нейросеть DeepSeek для написания кода. 
В приложении реализованы все функции в соответствии с ТЗ, а так же некоторые дополнительные, такие как: Мини-игры, возможность просмотра меню для заказа еды в приложении, а так же множество небольших улучшений, призваных отполировать приложение и добавить функции, которые необходимы в приложении, позиционирующем себя как user-friendly. Данные функции предоставляют пользователю большую вариативность действий (Возможность бронирования через чат-бота, выбор клуба для бронирования в нём пк через список клубов, а не только через соответствующий экран. Возможно просматривать доступные для брони Пк как в виде списка, так и на карте и т.д.), а так же большую информативность приложения (Интеграция внешних сервисов, автоматическая генерация ссылок на посты в ВК, встроенная карта города с координатами клубов, возможность автоматически связаться по контактам с каким-либо клубом через ТГ, телефон, почту и т.д.)

## Обзор

Приложение BBPlay — это клиент для компьютерного клуба, написанный на Flutter. Оно предоставляет функционал:
- просмотр новостей из VK,
- список клубов с контактами и картой,
- бронирование компьютеров и пакетов,
- управление личными бронированиями,
- поддержка (чат-бот на базе Gemini AI),
- профиль пользователя (баланс, редактирование, пополнение),
- мини-игры.

Приложение использует **архитектуру, приближенную к MVC/сервис-ориентированной**, без явного стейт-менеджмента (Provider/BLoC). Состояние экранов управляется через `setState`, данные между экранами передаются через `go_router` (`extra`), а долгоживущие данные сервисов хранятся как синглтоны.

## Маршрутизация и навигация

Используется `go_router` с двумя основными структурами:

- **GoRoute** для `/login` и `/register` – экраны без нижней навигации.
- **ShellRoute** – обёртка с `ScaffoldWithBottomNavBar`, содержащая дочерние `GoRoute`:
  - `/news` → `NewsScreen`
  - `/clubs` → `ClubsScreen`
  - `/booking` → `BookingScreen` (принимает `extra` параметры для бронирования)
  - `/support` → `SupportScreen`
  - `/profile` → `ProfileScreen`

**Redirect** проверяет авторизацию: если пользователь не залогинен и путь не `/login`/`/register` – перенаправляет на `/login`. Если залогинен и пытается зайти на логин/регистрацию – перенаправляет на `/news`.

**Передача параметров**:
- Из бота в экран бронирования: `context.go('/booking', extra: {...})`
- В `BookingScreen` параметры извлекаются и передаются в `BookingTab`.
- Для сохранения состояния между вкладками (например, выбранный клуб, дата, время, режим отображения) используется `BookingFormService` – синглтон с полями `cafeId`, `date`, `time`, `duration`, `viewMode`.

## Слои архитектуры

### 1. UI (представление)

- **Экраны (screens)** – `StatelessWidget` или `StatefulWidget`. Содержат постраничную логику, вызовы сервисов, обработку жестов.
- **Виджеты (widgets)** – переиспользуемые части, часто с собственным состоянием (например, карта зала, список ПК).
- **Вкладки в `BookingScreen`**: `BookingTab` (форма выбора и карта/список), `MyBookingsTab`, `GamesTab`.

### 2. Бизнес-логика и сервисы

Все сервисы реализованы как **синглтоны** (фабричный конструктор `_internal`). Они не зависят от Flutter-виджетов, что упрощает тестирование.

| Сервис | Назначение |
|--------|-------------|
| `AuthService` | Вход, выход, сохранение/загрузка пользователя (SharedPreferences). |
| `ApiService` | Все HTTP‑запросы к бэкенду (клубы, бронирование, цены, профиль, пополнение). |
| `RegistrationService` | Создание аккаунта, запрос/верификация SMS. |
| `SupportBotService` | Управление чатом с ботом: история, контекст, пошаговый опрос бронирования. Интегрирует `GeminiService`. |
| `GeminiService` | Отправка запросов к Google Gemini AI с системным промптом и актуальными данными клубов. |
| `VkApiService` | Запрос записей стены VK (сервис‑токен). |
| `BookingFormService` | Сохранение параметров бронирования (дата, время, клуб, длительность, режим просмотра). |
| `ClubDataService` | Статическая информация о клубах (цены, оборудование, контакты). Используется в контексте для бота. |

### 3. Модели данных (models)

Модели содержат фабричные методы `fromJson` для десериализации ответов API. Основные модели:

- `User` – авторизованный пользователь.
- `Cafe` – клуб (id, адрес).
- `PC`, `Room`, `Price`, `Product`, `UserBooking` – бронирование.
- `NewsPost`, `VideoAttachment`, `PollAttachment`, `LinkAttachment` – новости.
- `CreateMemberResponse`, `RegistrationResponse` – регистрация.

## Ключевые особенности реализации

### 1. Карта зала (`BookingMapWidget`)

- Загружает `struct-rooms-icafe` (список комнат и ПК).
- Использует `CustomPainter` для отрисовки комнат (прямоугольники с текстом) и ПК (цветные квадраты).
- Взаимодействие – `Positioned` + `GestureDetector` поверх каждого ПК.
- Масштабирование и панорамирование – `InteractiveViewer`.
- Координаты ПК и комнат корректируются хардкодными правками (комната VIP, GameZone).
- Кнопка «Пакеты» открывает модальное окно со списком продуктов.

### 2. Чат-бот с поддержкой (`SupportBotService` + `GeminiService`)

- **Пошаговый опрос бронирования**: бот последовательно запрашивает клуб, дату, время, длительность, затем формирует действие `BotAction.navigateToBooking`.
- **Контекст**: при инициализации сервис загружает клубы и цены, передаёт их в `GeminiService`.
- **История сообщений** хранится внутри сервиса, UI отображает её через `ListView`.
- **Индикатор печати** – анимированные точки.
- **Действия** – после ответа бот может предложить кнопку «Перейти к бронированию», которая сохраняет параметры и выполняет переход через `go_router`.

### 3. Новости (VK API)

- `VkApiService` вызывает `wall.get` с сервисным токеном.
- `NewsScreen` отображает карточки с текстом, фотогалереей, видео, опросами, ссылками.
- Фотогалерея – `PageView` с динамической высотой (рассчитывается по соотношению сторон).
- При нажатии на видео или ссылку – переход в браузер через `url_launcher`.

### 4. Регистрация

- Двухшаговый процесс: создание участника → запрос SMS → верификация кода.
- Используются отдельные эндпоинты `.../cafe/.../members`, `/request-sms`, `/verify`.
- После успешной верификации пользователь перенаправляется на экран входа.

### 5. Профиль и пополнение баланса

- `ProfileScreen` загружает данные через повторный логин (`AuthService.login`).
- Поля телефона и email можно редактировать через API `updateMemberProfile`.
- Пополнение баланса – через `ApiService.topUpBalance` (вызов `.../members/action/topup`).
- Выход – очистка `SharedPreferences` и переход на `/login`.

### 6. Бронирование (вкладки)

- **BookingTab**:
  - Выбор клуба (Dropdown), даты (DatePicker), времени (TimePicker), длительности (Dropdown).
  - Переключение режимов: список (PcListWidget) / карта (BookingMapWidget).
  - Проверка доступности (`ApiService.getAvailablePCs`).
  - Отображение цен (`ApiService.getPrices`).
  - Модальные окна с деталями ПК или пакета.
- **MyBookingsTab**:
  - Список бронирований через `ApiService.getUserBookings`.
  - Отмена бронирования через `/booking-cancel`.
- **GamesTab**:
  - Три простые игры (угадай число, камень‑ножницы‑бумага, крестики‑нолики) в модальных окнах.

### 7. Сохранение состояния формы бронирования

`BookingFormService` (синглтон) сохраняет:
- `cafeId`, `date`, `time`, `duration`
- `viewMode` (list/map)

При переключении между вкладками (`BookingTab` → `MyBookingsTab` → обратно) или при переходе из бота значения восстанавливаются.

## Потоки данных

1. **Авторизация**  
   `LoginScreen` → `AuthService.login` → токен/пользователь сохраняются → `go_router` перенаправляет на `/news`.

2. **Бронирование через UI**  
   `BookingTab` → выбор параметров → `ApiService.checkAvailability` → отображение доступных ПК → выбор ПК/пакета → `ApiService.createBooking`.

3. **Бронирование через бота**  
   `SupportScreen` → `SupportBotService.sendUserMessage()` → пошаговый сбор параметров → `BotAction` → сохранение параметров в сервисе → `context.go('/booking', extra: params)` → `BookingScreen` принимает параметры и передаёт в `BookingTab`.

4. **Новости**  
   `NewsScreen.initState()` → `VkApiService.fetchWallPosts()` → парсинг моделей → отображение.

5. **Профиль**  
   `ProfileScreen` → `AuthService.login` (обновление) → отображение данных → при редактировании `ApiService.updateMemberProfile` → повторная загрузка.

## Взаимодействие с бэкендом

Базовый URL: `https://vibe.blackbearsplay.ru`

Основные эндпоинты:

| Метод | Путь | Назначение |
|-------|------|-------------|
| GET | `/cafes` | список клубов |
| GET | `/struct-rooms-icafe` | структура зала |
| GET | `/available-pcs-for-booking` | доступные ПК на выбранное время |
| POST | `/booking` | создание брони |
| POST | `/booking-cancel` | отмена брони |
| GET | `/all-prices-icafe` | цены и пакеты |
| GET | `/all-books-cafes` | брони пользователя |
| POST | `/login` | авторизация (возвращает `private_key`) |
| POST | `/api/v2/cafe/{cafeId}/members` | создание участника |
| POST | `/request-sms` | запрос SMS‑кода |
| POST | `/verify` | верификация кода |
| POST | `/api/v2/cafe/{cafeId}/members/action/topup` | пополнение баланса |
| PATCH | `/api/v2/cafe/{cafeId}/members/{memberId}` | обновление профиля |

Все запросы используют `Content-Type: application/json`. Коды успеха: `0`, `200`, `201`. Ошибки выбрасываются как `Exception`.

## Управление состоянием

- **Локальное состояние виджетов** – `setState`.
- **Глобальное состояние сессии** – `AuthService` (синглтон + SharedPreferences).
- **Межэкранное состояние формы бронирования** – `BookingFormService` (синглтон).
- **История чата** – внутри `SupportBotService`.
- **RefreshIndicator** для обновления списков.

Отсутствие внешнего стейт-менеджмента (Provider/BLoC) оправдано небольшим размером приложения и простотой взаимодействия между экранами через сервисы-синглтоны.

## Заключение

Архитектура BBPlay представляет собой грамотно организованное Flutter-приложение с чётким разделением на экраны, виджеты, сервисы и модели. Ключевые решения:

- `go_router` для декларативной навигации и защиты маршрутов.
- Сервисы-синглтоны для бизнес-логики и API.
- `BookingFormService` для сохранения состояния между вкладками.
- Чат-бот с пошаговым опросом и интеграцией Gemini AI.
- Кастомная отрисовка схемы зала через `CustomPainter`.

Приложение готово к масштабированию: при необходимости можно внедрить реактивное управление состоянием (Riverpod/BLoC) и вынести HTTP-слой в отдельный репозиторий, но текущая архитектура уже обеспечивает хорошую поддержку кода и понятность.
